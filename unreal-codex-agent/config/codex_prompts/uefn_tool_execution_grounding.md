# UEFN Tool Execution Grounding

Use this as execution grounding for chat actions across all model providers.

## Tool-first rule

- If the user asked for a change, do not stop at a text answer when a tool or server action can execute it.
- Prefer first-class server actions and registered tools before generic prose.
- When the request is for a build or model, route through the planner-backed build actions first so geometry is generated by code instead of narrated in text.

## Action selection

- Use `build_house_action` for house-like requests.
- Use `build_structure_action` for non-house structures that still need coherent geometry.
- Use `terrain_action` for terrain and landform requests.
- Use `import_attached_models` when importable FBX or ZIP attachments are part of the request.
- Use `run_uefn_tool` when a catalog tool already covers the operation precisely.
- Use `execute_python_in_uefn` only when the existing actions/tools cannot do the job cleanly.
- If the exact low-level tool is unclear, consult the tool search/detail helpers and then execute the matching tool rather than answering vaguely.

## Quality rule

- After an edit, verify it with live UEFN context or a follow-up query before claiming success.
- Avoid generic "here is what you could do" replies when the request is executable.
- Prefer concrete actions, managed reuse, and deterministic support-aware placement.
- If attachments contain importable models, import and place them before inventing replacement primitives.
